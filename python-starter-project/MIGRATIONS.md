# Database Migration Strategy

Although the template does not ship with a database, this guide standardizes how to introduce one.

## Tooling Options
- **Gold standard:** `alembic` (SQLAlchemy ecosystem) with autogenerated revisions and offline migrations.
- **Lightweight fallback:** Plain SQL files applied via `psql` or `pgcli` wrapped in Makefile tasks.

## Workflow
1. Update ORM models / schema definitions.
2. Generate migration (`alembic revision --autogenerate -m "add users table"`).
3. Review generated SQL, adjust for idempotency.
4. Run migrations locally (`alembic upgrade head`).
5. Commit migration scripts alongside feature branch.
6. CI runs migrations against ephemeral database (TODO pipeline integration).
7. Deploy: apply migrations prior to releasing new application image.

## Verification
- Run smoke tests hitting new schema.
- Validate rollback plan (e.g., `alembic downgrade -1`).
- Monitor database metrics and error budgets.

## Red-Team Box
- **Risk:** Irreversible destructive migrations in production. Mitigation: require peer review and staging rehearsal.
- **Risk:** Long-running migrations causing downtime. Mitigation: prefer `CONCURRENT` indexes, chunked backfills.
- **Risk:** Drift between environments. Mitigation: enforce migrations in CI for every PR.
